## 三、图片 URL 格式约定

项目中使用的图片 URL 格式:

| 格式 | 示例 | 说明 |
|------|------|------|
| `local:xxx` | `local:img_123456` | 本地 IndexedDB 存储 |
| `data:...` | `data:image/png;base64,xxx` | Base64 内联 |
| `http://...` | `http://example.com/img.png` | HTTP 图片 |
| `https://...` | `https://example.com/img.png` | HTTPS 图片 |
| `blob:...` | `blob:http://localhost:3000/xxx` | 临时 Blob URL |
| `video:xxx` | `video:vid_123456` | 本地视频存储 |

## 四、问题分析

### 4.1 重复实现严重

| 函数 | 出现位置 |
|------|---------|
| `resolveImageUrl/resolveImageSrc` | 5+ 处 |
| `blobToBase64` | 5+ 处 |
| 图片解析逻辑 | 7+ 处 |

### 4.2 逻辑不一致

不同位置的实现细节不同:
- 有的处理 `local:`，有的处理 `blob:`
- 有的返回 base64，有的返回 blob:
- 错误处理方式不同

### 4.3 getImageDimensions 的问题

当前实现 (canvasIntegrationService.ts L88-97):
```typescript
async function getImageDimensions(imageUrl: string) {
  return new Promise((resolve, reject) => {
    const img = new Image();
    img.onload = () => resolve({ width: img.naturalWidth, height: img.naturalHeight });
    img.onerror = reject;
    img.src = imageUrl;  // ⚠️ 网络 URL 可能因 CORS 失败
  });
}
```

**问题**:
- 网络 URL（http/https）如果没有 CORS 头，`new Image()` 会加载失败
- CORS 错误无法被 `onerror` 捕获详细信息

## 五、整合建议

### 5.1 新建统一图片服务

建议创建 `services/unifiedImageService.ts`:

```typescript
export const unifiedImageService = {
  // 解析任意格式图片 URL，返回可显示的 URL
  resolveForDisplay: async (url: string): Promise<string>
  
  // 解析任意格式图片 URL，返回 Base64
  resolveForApi: async (url: string): Promise<string>
  
  // 获取图片尺寸（兼容所有格式）
  getDimensions: async (url: string): Promise<{ width: number, height: number }>
  
  // 保存图片到本地存储
  saveToLocal: async (id: string, blob: Blob): Promise<void>
  
  // 从本地存储读取
  getFromLocal: async (id: string): Promise<Blob | null>
  
  // 解析 URL 类型
  parseUrl: (url: string): ImageSource
}
```

### 5.2 统一 getDimensions 实现

```typescript
async function getDimensions(url: string): Promise<{ width, height }> {
  // 1. 如果是 base64 或 blob，直接用 new Image()
  // 2. 如果是 local:，先从 IndexedDB 读取 blob，再用 createObjectURL
  // 3. 如果是网络 URL，用 fetch + createObjectURL 绕过 CORS
}
```

### 5.3 逐步迁移

1. 第一阶段: 创建统一服务
2. 第二阶段: 在新代码中使用统一服务
3. 第三阶段: 逐步重构旧代码
